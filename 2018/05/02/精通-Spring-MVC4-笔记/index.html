贫血的领域模型

Eric Evans出色的书籍Domain Driven Design对这些图层有以下说明。

应用层：定义软件应该做的工作，并指导表达域对象来解决问题。该层负责的任务对业务有意义或者与其他系统的应用层进行交互所必需的。这层保持薄。它不包含业务规则或知识，只是协调任务并将工作委托给下一层下的域对象的协作。它没有反映业务情况的状态，但它可以具有反映用户或程序的任务进度的状态。
域层（或模型层）：负责表示业务概念，业务情况信息和业务规则。反映业务情况的状态在此处受到控制和使用，即使将其存储的技术细节委托给基础架构。这一层是商业软件的核心

Eric EvansDomain Driven Design

其中有一项核心的理念就是将面向对象的范式应用到领域对象之中。如果违背这一原则的话，就会被称之为贫血的领域模型（Anemic Domain Model）。
详细可以查看Martin Fowler的博客。
如何避免领域贫血的途径如下：

  1. 服务层合适进行应用级别的抽象（如事务处理）而不是业务逻辑；
  2. 领域对象应该始终处于合法状态。通过校验器（validator）或JSR-303的校验注解，让校验过程在表单中进行；
  3. 将输入转化成有意义的领域对象；
  4. 将数据层按照Repository的方式来实现，Repository中会包含领域查询（参考Spring Data规范）；
  5. 将领域逻辑和底层的持久化解耦；
  6. 尽可能使用实际的对象，例如操作FirsName类而不是操作String。
